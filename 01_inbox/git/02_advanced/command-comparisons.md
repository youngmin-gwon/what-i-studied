## Git 주요 명령어 비교: 비슷해 보이지만 다른 결정적 차이

Git 에는 결과는 비슷해 보이지만 작동 원리와 히스토리에 미치는 영향이 완전히 다른 명령어들이 있습니다. 이 차이를 명확히 아는 것이 숙련자와 초보자를 가르는 기준입니다.

---

### 1. Merge vs Rebase (병합 기술)

둘 다 두 브랜치를 합치는 도구이지만, **히스토리의 형태**가 달라집니다.

| 특징 | Merge | Rebase |
|:---|:---|:---|
| **히스토리** | 비선형 (병합 커밋 생성, 가지가 남음) | 선형 (커밋을 재배치하여 한 줄로 만듦) |
| **안전성** | 안전함 (기존 커밋을 변경하지 않음) | 주의 필요 (기존 커밋의 해시가 변경됨) |
| **사용 시점** | 공용 브랜치에 기능을 합칠 때 | 내 로컬 작업을 최신 상태로 맞출 때 |
| **복구** | 쉬움 (병합 커밋만 지우면 됨) | 어려움 (`reflog` 필요) |

>[!IMPORTANT] **Golden Rule**
>이미 서버에 push 된 공용 브랜치(main, develop)는 절대 **Rebase** 하지 마세요. 다른 팀원들의 히스토리가 완전히 깨집니다.

---

### 2. Reset vs Revert (되돌리기 기술)

과거로 돌아가는 방식의 차이입니다.

| 특징 | Reset | Revert |
|:---|:---|:---|
| **작동 방식** | 시간을 되돌림 (커밋 히스토리를 삭제) | 거꾸로 가는 커밋을 추가함 |
| **원격 영향** | `force push` 가 필요함 | 일반 `push` 가능 (안전) |
| **협업** | 나 혼자 쓰는 브랜치에서만 사용 | **공용 브랜치에서 필수 사용** |
| **흔적** | 히스토리에서 사라짐 | "Revert '…'"라는 기록이 남음 |

---

### 3. Switch vs Checkout (이동 기술)

Git 2.23 버전부터 너무 많은 기능을 가진 `checkout` 의 역할을 분산시키기 위해 도입되었습니다.

- **Check out**: 브랜치 전환, 파일 복구, 특정 커밋 이동 등 혼란스러운 멀티 플레이어.
- **Switch**: 오직 **브랜치 전환**에만 집중. (더 직관적임)
- **Restore**: 오직 **파일 복구**에만 집중.

---

### 4. Fetch vs Pull (가져오기 기술)

- **Fetch**: 원격의 최신 정보를 로컬로 '가져만 오기'. (내 코드는 안 바뀜)
- **Pull**: 원격 정보를 가져와서 내 코드와 '병합하기'. (`fetch` + `merge`)

---

## 🚨 흔한 실수 (Common Mistakes)

1. **`git pull` 만 믿고 작업하기** ❌
   - `pull` 은 자동으로 병합을 수행하므로 원치 않는 머지 커밋이 남을 수 있습니다. 깔끔한 히스토리를 원한다면 `git pull --rebase` 를 생활화하세요.
2. **`reset --hard` 후 커밋 유실**
   - 앞서 설명했듯, `reset` 은 히스토리를 지웁니다. 백업이 없다면 위험합니다.
3. **`checkout` 과 `switch` 혼용**
   - 최신 Git 을 사용한다면 목적이 명확한 `switch` 와 `restore` 를 사용하는 것이 실수를 줄이는 길입니다.
